## 一、微信分享预览机制归档（659ccd8 的背景与取舍）

微信（及部分国内 IM/浏览器）的分享抓取器**不读 Open Graph 协议**，其缩略图来源按大致优先序为：

1. 站内 JS-SDK 主动声明（`wx.config` + 分享接口）——需公众号资质与后端签名，与本站"零依赖纯静态"约束冲突，**不做**；
2. `<link rel="image_src">` ——传统站长协议，部分抓取器读取；
3. 页面正文中的首批可见 `<img>` ——多数微信内置浏览器分享走此通道，且**隐藏图（display:none 等）会被跳过**。

因此 659ccd8 的低成本兼容方案为双通道：

- `<head>` 补 `<link rel="image_src" href="…/assets/og/og-card.png">`（绝对 URL，属约定特批的分享协议头例外；页面内资源仍一律相对路径）；
- 首页尾部新增**可见**站铭卡 `<figure class="brand-card"><img src="assets/og/og-card.png" …>`，即 og 同源图的正文实体，供只抓正文图的爬虫命中；配文承担"邀请分享"职能，故不隐藏、不移出首屏文档流。

局限（当时已知，此处归档备查）：微信对个人 GitHub Pages 域名的抓取存在缓存与白名单不确定性，实际预览效果以真机分享为准；若日后更换仓库名/域名，`head` 内两处绝对 URL 与 og:url/og:image 需同步修改。

## 二、勘误：旧文档"内嵌走抽屉"描述已被推翻

`docs/delivery_vision_round6.md` §三记载"**<680px 或全屏地图态下，点地点改为底部滑出抽屉**"。该行为在 Xiangtao 真机复测中发现与页面级缩放冲突（fixed 抽屉在放大态只露一角），已于 659ccd8 修正，**以下为现行口径**：

- **内嵌地图（含窄屏）**：点击地点展开为地图下方**文档流卡片**（窄屏轻滚 `block:nearest` 使其入视野，不与用户缩放状态冲突）；
- **全屏放大态**：维持底部滑出抽屉（四通道关闭、焦点管理不变）；
- 切换人物时地点卡重置为初始提示。

旧文档按 conventions v1.6"旧文件不改名（不迁移）"原则保留原文，阅读时以本追记为准。r7 轮播放聚焦改造（字幕条）建立在上述现行口径之上。
