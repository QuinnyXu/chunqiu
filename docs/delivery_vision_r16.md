# 交付说明 · Vision r16

**任务**：修复关系图「放大查看」全屏态两处 bug——(a) 桌面全屏内点边/点节点无反应；(b) 手机全屏视图无显著放大且点边无反应。统一到「地图全屏」的既有交互模式（fit-to-container 取景、双指缩放/拖移、底部抽屉三件套关闭）。

**改动文件**：仅 `site/app.js`（无 CSS/HTML 改动——底部抽屉 `#place-drawer` 的桌面限宽 520px 居中 / 手机全宽样式、及下滑/遮罩/X/ESC 三件套此前已为地图全屏建好，本轮直接复用）。

## 一、根因分析（对领队此前假设的确认与修正）

领队此前提出两个排查方向：①事件绑定绑到了错误的 DOM 节点；②全屏容器尺寸初始化时为 0 导致 fit 失败。实测排查结论如下：

### 根因①：克隆丢失事件绑定（对应「点边/点节点无反应」，桌面＋手机同源）

- **不是「绑到错误节点」，而是「克隆体上根本没有绑定」。** `openRelOverlay()` 进入全屏时用 `src.cloneNode(true)` 把内嵌关系图 SVG 克隆进浮层。`cloneNode(true)` **只复制 DOM 结构与属性，不复制经 `addEventListener` 注册的事件处理器**。而内嵌图的点边/点节点交互（`drawEgoGraph`/`drawPanoGraph`/`egoNodeEl` 里 `path.addEventListener("click", …)`、`g.addEventListener("click", …)`）全部是直接绑在各 SVG 元素上的——克隆出来后这些元素成了「哑元素」，点击毫无反应。
- 这解释了为何桌面与手机**同时**无反应：与指针类型、指针捕获（r10 那次桌面 bug）都无关，纯粹是克隆体没有处理器。内嵌态两端（桌面右侧卡片、手机）用的是原始未克隆的图，处理器俱在，故正常。
- **修复**：绘图时给每条边（`path` 与多重关系数字徽记 `badge`）打 `data-detail` 索引、给每个节点（含中心/家系/侧组）打 `data-node`＝人物 id，并把 `pair.rels` 存入每轮重建的 `relView.detailReg` 注册表；`openRelOverlay` 克隆后调 `rebindRelClone(clone)`，按这两个 data 属性在克隆体上**重新绑定 click**。且全屏内一律路由到**底部抽屉**（`showRelDetailDrawer`），而非内嵌态的右侧卡片。误触由 `relZoom.panDist > 6` 守卫（与地图锚点同规则，拖移后不算点击）。

### 根因②：取景是「contain 缩得偏小」，非「容器尺寸为 0」（对应手机「视图无显著变化」）

- **修正领队假设**：旧代码的 fit **根本没有读容器尺寸**（只用图形 `getBBox()` 与图自身 viewBox 比例算 contain 取景），所以「容器尺寸为 0」并非当前症状的成因。`getBBox()` 取的是几何外接框、与容器像素尺寸无关，也不会因容器 0 而失败。
- **真正成因**：旧 fit 把整张图「contain」进 viewBox（保证整图可见）。关系图（尤其 ego 图）**又宽又矮**，在**竖屏手机**上，contain 取景被宽度卡死——渲染缩放 ≈ 内嵌态的 `容器宽/图宽`，与内嵌几乎一样大，上下大片留白，故「看着没放大、还是很小」。这是取景策略问题，不是尺寸读取失败。
- **修复**：改为真正的 **fit-to-container**。进入全屏时实测浮层容器 `clientWidth/Height`：
  - **竖屏容器（手机全屏，高>宽）**：viewBox 纵横比取**容器比例**、做 **cover** 取景（填满容器、裁掉过宽的两侧，拖移浏览）。经数学模拟：ego 图放大约 **3.7×**、pano 图约 **2.5×**，内容填满屏幕高度，明显可读。
  - **横屏/桌面容器**：取**图自身比例**做 **contain**（整图可见、不裁切），维持既有桌面观感、**零回归**（桌面本就不小，且报告的桌面 bug 只是点击无反应）。
  - 纵横比一经进入全屏定死（存 `relZoom.aspect`），拖移/滚轮/双指缩放全程沿用，避免首次手势跳变；`zClamp` 与手势 pinch/wheel 同步改用该比例。
- **容器 0 兜底**：虽非当前成因，仍按领队提示加护栏——若本帧 `clientWidth/Height` 恰为 0（极端布局时序），fit 安全回退到「图自身比例 contain」（永不比旧行为更差），并 `requestAnimationFrame` 下一帧再量一次补正。

## 二、全屏抽屉（对齐地图全屏）

- 点边 → 抽屉标题「A — B（n 重关系）」/「这条关系」；点节点 → 「X 的一度关系（n）」。
- 抽屉正文由 `relDetailBody(rels, pid)` 生成，**与内嵌右侧卡片同一函数**，确保内容严格一致：多重关系全部列出、`rel_label`、可靠度、依据（`source_note`）、姓名行与小传。
- 复用 `#place-drawer`：桌面限宽 `min(520px, 100vw−2rem)` 居中浮卡、手机全宽贴底；关闭三件套四路——X 按钮 / 点遮罩外部 / 下滑手势 / ESC，全部沿用既有全局绑定，与地图一致。抽屉层级（z60）＋遮罩（z58）压在浮层（z50）之上，开启时挡住图、防误拖。
- `closeRelOverlay()`/`closeOverlay()` 关闭全屏时若抽屉仍开则一并收起，避免孤悬（顺带修了地图侧同类潜在遗留）。

## 三、内嵌态无退化

- 内嵌图仍用原始（未克隆）SVG 与原 `addEventListener` 处理器：桌面点边/点节点 → 右侧卡片、手机同此，行为不变。新增的 `data-detail`/`data-node` 属性对内嵌交互无副作用（内嵌不走 rebind）。
- 全景「仅主角边」过滤、ego 折叠展开、面包屑游走、以人为中心重绘等内嵌功能均未触及。

## 五、验收偏差上报

- **无阻断性偏差。** 一处设计取舍据实说明：竖屏手机对 pano（环形）图做 cover 取景时，居中裁宽会先展示环的中段、左右两侧节点需拖移查看——这是「又大又满的环形图落在窄竖屏」的固有取舍，与「明显放大＋可拖移浏览」的验收目标一致，非缺陷。ego 图中心即中心人物，裁宽后正见主角，无此顾虑。
- 本环境浏览器扩展（Claude in Chrome）不可用，未能跑真机点按录屏；已用语法校验＋取景数学模拟＋逐交互路径代码复核替代。若 Xiangtao 真机复测发现细颗粒问题（如极窄图的边线命中区偏小），可另起微调轮次。
