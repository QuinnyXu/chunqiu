## 1. 目录结构与数据流

```
data/csv/          唯一数据源（source of truth），所有编辑发生在这里
data/incoming/     待合入的增量数据；复核通过后手工合入 data/csv/ 并删除
tools/             管线脚本（仅 Python 标准库）
site/              静态站根目录，可整体部署到 GitHub Pages
site/data/         生成的 JSON —— 生成物，禁止手改
site/assets/       图标（icons/）与地图底图（map/）
docs/              本文件与设计文档（design/）
private/           私有层（写作提示等，git 忽略，不入公开仓库）
```

数据流（单向）：

```
data/csv/*.csv  →  tools/csv_to_json.py  →  site/data/*.json + meta.json
```

- **JSON 禁止手改**：`site/data/` 下任何文件的改动只能来自重跑 `python tools/csv_to_json.py`。
- 每次改动 `data/csv/` 后的标准流程：`python tools/validate.py` 通过 → `python tools/csv_to_json.py` 重新生成 → 一起提交。

## 2. ID 规范

| 表 | 前缀/格式 | 示例 | 说明 |
|---|---|---|---|
| events | `E###`（可带一位大写字母后缀） | E010, E005A | **接续现有编号**（当前已用到 E020）；新事件尽量用纯 `E###`，字母后缀仅用于向既有编号之间插入事件 |
| people | `P_XXX`（大写拼音） | P_WENJIANG | 见下方「谥号形近加 GONG 后缀」通则 |
| places | `L_XXX`（大写拼音） | L_LINZI | 与下方 sources 的 `L###`（穀梁传）命名空间不同不冲突，见说明 |
| sources | `Z###` 左传 / `S###` 史记 / `G###` 国语 / `A###` 考古 / `P###` 诗经 / `B###` 现代研究 / `Y###` 公羊传 / `L###` 穀梁传 / `T###` 先秦诸子及秦汉说部层 | Z001, S002, Y001, T001 | 九个前缀均合法，按来源性质归类；`Y`/`L` 为 v1.9 新增（春秋三传补齐第二、三家）；`T` 为 v1.10 新增，见下方「T 前缀分层纪律」 |
| passages | `Q###`（可带后缀） | Q004A | |
| background | `BKG###` | BKG001 | |
| archaeology | `ARC###` | ARC001 | |
| relations | `R###` | R001 | 人物关系结构表，供图谱用 |

ID 一经使用不得复用或改义；删除记录时其 ID 作废封存。

**ID 退役**：作废 ID 永不复用，也不得赋予新含义；引用它的历史提交以 git 历史为准。现有退役名单：**E005、E006**（已拆分为 E035/E036 与 E037–E039）。

**sources 的 `L###`（穀梁传）与 places 的 `L_XXX`（地点）不冲突**：前者是「`L` + 三位数字」（如 `L001`），后者是「`L_` + 下划线 + 大写拼音」（如 `L_XIAO`），正则形态不同、`validate.py` 分属两张表独立校验，不会互相误判。

**谥号形近加 GONG 后缀通则（v1.9）**：新人物 ID 若与库内既有人物 ID 仅差极少字符、易生混淆（尤其是同姓不同代国君，如"晋献公"与"晋襄公"），命名时在**较晚出现或本轮新入库的一方**后缀加 `GONG`（取"公"字拼音，与"侯/伯"等其他称谓无关，纯作辨识后缀）以显式拉开差异，不满足于"仅差一字母"的脆弱区分。范例：round13 新入库晋襄公本拟 `P_JINXIANG`，与既有 `P_JINXIAN`（晋献公）仅差一个 `G`、relations/event_people 中二者又同时出现，极易看串（Sophia 已在 round13 CHANGES.md 中主动预警），领队裁定合入前将新 ID 改为 **`P_JINXIANGGONG`**，`P_JINXIAN` 不动。后续遇到类似形近场景（如未来若入库"晋景公"与已有别的"晋X公"形近字符重叠），优先套用本通则加后缀，而非改动既有 ID。

**`T###` 前缀分层纪律（v1.10，裁定4）**：`T` 前缀专用于**先秦诸子及秦汉说部层**史料——即诸子（《韩非子》《说苑》《吕氏春秋》等）中的寓言、说理性举例，与秦汉说部/杂史中的轶闻叙事，**不是**《左传》《国语》一类当代编年记事，也不同于《史记》一类"后出叙事"的正史体例，故单立前缀不与 Z/S 混编。启用 `T` 前缀的条目须遵守：
- **严格分层**：`passages.quote_type` 一律标"经义异闻"或后续如为该层专设新枚举值，不得标"原文"；`events.reliability` **从严默认标 `low`**（诸子寓言/说部叙事常以理服人或渲染效果为先，纪年、人物细节多不可考，除非有独立经传/考古佐证方可上调），不得因文笔生动或情节完整而误标 high/medium。
- **不得单独立事件骨架**：`T` 类史料只作为已有经传骨架事件（`Z`/`S` 等来源支撑的 `events` 行）的旁证摘录挂入 `passages`，不得仅凭诸子/说部材料**新建**一条 `events` 行——避免寓言、说理故事被误当作编年史实收入主线。
- **来源标注**：`sources.notes` 须注明其"诸子说理"或"说部轶闻"性质及成书年代层级（先秦诸子 / 秦汉杂史），供合入者与读者一望而知非当代记事。
- 首次启用参见 round15（Sophia，楚庄王线，"绝缨之宴"《说苑》、"一鸣惊人"《韩非子》等）：round15 本轮仅备料、不合入，但本条规则与下方 `validate.py` 正则已提前落地，届时合入不会卡在校验上。

## 3. 事件分类枚举（events.category）

共 **15** 个，图标一一对应（`site/assets/icons/` 下按同名维护）：

```
即位, 战争, 会盟, 相会, 婚嫁, 生育, 出奔, 弑杀, 薨卒, 丧葬, 外交, 内乱, 灾异, 礼俗, 其他
```

「相会」与「会盟」的区分：「会盟」指诸侯/国家层面的正式会、盟、朝聘；「相会」指个人之间的私会或非诸侯会盟性质的相见（主要用于文姜与齐襄公一线）。

- 每个事件只取一个分类；复合事件取对主线叙事最核心的一面，拿不准时在交付说明中列出待复核。
- `importance`：`1`=主线关键，`2`=重要，`3`=背景。

## 5. reliability / certainty 三级标注

适用于 `events.reliability`、`places.certainty`、`places.coord_certainty`、`background.certainty`、`archaeology.certainty`：

- `high`：可作为骨架直接使用。
- `medium`：宜复核后使用。
- `low`：只适合作线索，不可直接当事实。

补充提醒（同旧库）：《诗经》条目是传统阐释和舆论层材料，不直接等同于编年史实；考古表目前是写作型摘要，学术级使用需补发掘报告出处。

## 6. CSV 编辑规约

- 编码 UTF-8，逗号分隔，首行表头；单元格内多值用 `;` 分隔（如 `source_ids`）。
- `year_bce` 为**负整数**表示公元前（前694 → `-694`），有效范围 **[-800, -480]**（v1.11 放宽，原 [-800,-600] 已不敷春秋中晚期史事——楚庄王卒于前591、邲之战前597 皆晚于旧上限；新上限前480 覆盖传统春秋下限前481/前476 一带，为该期主角入库的前置 schema 调整，领队裁定，本轮仅边界放宽、无数据变更）。
- 旧数据行的新增列允许留空；空值一律用空字符串，不写 `NULL`/`N/A`。
- 新增数据先进 `data/incoming/`，复核后合入 `data/csv/`。
- `event_people.presence`：枚举 `{亲至, 相关}`，允许暂空（**空视同亲至**）。亲至=此人当时身在事发地；相关=事件与其相关但本人不在场（如为其议婚、他人代行）。
- `events.sort_key`：可空整数；同一 `year_bce` 内按 sort_key 升序排列，按经传季月定值，建议以 10、20、30 间隔留插空。同一年内非空 sort_key 不得重复。

## 6.5 relations 表（v1.5）

表头 `id,person_a,person_b,rel_type,rel_label,reliability,source_note`。rel_type 枚举 8 类：`亲属-直系, 亲属-同辈, 婚姻, 君臣, 拥立, 敌对, 师友, 其他`。方向约定：**person_a 是 rel_label 的主语**（a=郑庄公、b=郑昭公、label=父）。每条关系只录一行、不录反向重复；同一对人物允许多行但 rel_type 必须不同。异说关系降 reliability 并在 source_note 说明。`people.relations` 自由文本列保留（人读友好），结构表供图谱等程序使用。

## 7. 私有层与叙事分层（v1.3）

- **私有层**：一切写作向列（历史上以 novel 开头命名的两个备注列，及后续任何写作向字段）不入公开数据，统一存 `private/writing_notes.csv`（通用三列 `table,row_id,note`；`private/` 整目录 git 忽略）。校验器对公开表头设有护栏：任何以 novel 开头的列名直接报错。增量包如有写作向备注，写在 CHANGES 附注区由合入者转存私有文件。
- **presence 从严通例**：凡"师行而君亲在无文"（某师伐某、某人灭某）、"使者代行"（使贼、使宛）、"称疾不往/不从"及追叙关联者，一律标「相关」；只有史文明书其人在事发地（会、盟、战、朝、奔、生卒于其地）方标「亲至」。宁严不滥。
- **后出叙事分层通例**：《史记》及更晚文献中戏剧化情节（射钩佯死、曹沫劫盟、彭生化豕之类），须在 summary 或摘录 quote_type 标明"后出叙事"层，不得与经传记事混写；科普展示面引用时保留分层标注。

## 8. 交付文档命名规范（v1.6）

各角色每轮工作结束后的交付说明，统一存于 `docs/`，命名格式：

```
delivery_<agent>_r<团队轮次>[a|b].md
```

- `<agent>`：角色小写英文名（如 `skipper`、`sophia`、`vision`）。
- `<团队轮次>`：团队协作的整体轮次编号（纯数字，不加前导零）。
- `[a|b]`：可选后缀，仅当同一轮次内该角色产出多份交付文档需要区分先后/子任务时使用。
- 示例：`delivery_skipper_r7.md`、`delivery_sophia_r7a.md`、`delivery_sophia_r7b.md`。

**旧文件不改名**：本规范生效前已存在的交付文档（如 `docs/delivery_vision_round4.md`、`docs/delivery_skipper_round1.md` 等 `round<N>` 命名）保留原名，不做迁移；仅自本规范生效起的新交付文档按上述格式命名。

## 9. 前端本地存储与外部资源例外（v1.7）

站点定位是纯静态展示，此前一律不落地任何浏览器本地存储、不引入站外脚本/资源。r13/r14 起出现实际需求（首访引导需记忆、访问统计需官方 beacon），本节明确解禁范围，避免"能不能用"每次临时判断。

### 9.1 localStorage 解禁范围

允许使用 `localStorage`，但**仅限**以下两类用途，超出即视为违规：

1. **首访标记类**：一次性提示/引导是否已看过（如首访三步引导 `TOUR_KEY`），只存布尔/字符串标记，不存用户输入内容或可识别个人信息。
2. **界面偏好类**：纯展示层的用户选择记忆（如列表/地图视图切换偏好、某折叠面板的展开状态），刷新后免于用户重新操作一次，不影响数据正确性。

**明确禁止**：不得用 localStorage 存放数据表内容或其缓存（数据以 `site/data/*.json` 为唯一运行时来源，避免本地缓存与线上数据不一致）、不得存放可识别用户身份的信息、不得用作分析/埋点的自建载体（埋点需求走 9.2 的官方 CF Analytics beacon，不自制）。

**读写规约**：所有 localStorage 读写必须包一层 `try/catch`（隐私模式/存储禁用下 `localStorage` 可能抛错），失败时静默降级为"当次会话不记忆"，不得让页面因此报错或白屏。参见 `site/app.js` 的 `tourSeen()`/`tourMarkSeen()` 写法。

### 9.2 外部资源例外清单

站点默认原则仍是"页面内资源一律相对路径，不请求站外"（`site/index.html` 头部注释已有说明），以下三项为**领队特批的固定例外**，除此之外新增任何站外请求（脚本、字体、图片、iframe 等）须重新走特批，不得自行比照追加：

| # | 用途 | 落地位置 | 备注 |
|---|---|---|---|
| 1 | og/twitter 分享协议图 | `site/index.html` 头部 `og:image`/`twitter:image`/`image_src`，指向 `https://chunqiu.timechorus.com/assets/og/og-card.png` | 分享协议要求绝对 URL，爬虫（微信等）不读相对路径；图片本身仍是仓库内资源，只是引用方式用了绝对地址，非真正的第三方资源 |
| 2 | Cloudflare Web Analytics（beacon） | `site/index.html` `</body>` 前一行 `<script defer src="https://static.cloudflareinsights.com/beacon.min.js" data-cf-beacon='{"token":"..."}'>` | 隐私友好、无 cookie 的官方访问统计脚本；`token` 明文出现在页面源码是 CF 官方设计如此，不构成信息泄露，无需混淆或后端代理 |
| 3 | 打赏/支持链接 | 关于页「版本与反馈」节，安静一行文案 | 链接目标（爱发电或 GitHub Sponsors）由 Xiangtao 提供；到位前页面留占位文案「支持链接（筹备中）」，不外链、不做假链接 |

三项之外，任何 CDN 脚本、外部字体、第三方 iframe 等一律不引入，保持站点静态自持、可离线审计的特性。

## 10. 协作节奏：备料与合入分离（v1.7）

自 round9 起团队实践稳定为"备料"与"合入"两个独立动作，本节成文固化，避免新人/新 agent 误以为交付即生效：

- **备料（Sophia）**：史料增量一律先落 `data/incoming/round<N>/`（含 `*_new.csv` 与 `CHANGES.md`），并在临时副本上完成"合并模拟"（拷贝主表 → append 全部增量 → 跑 `tools/validate.py` → 确认 exit 0），但**不动 `data/csv/` 主表**。合并模拟通过只代表"这批数据本身自洽、可合入"，不代表"已经合入"。
- **合入（Skipper）**：收到备料后按 `CHANGES.md` 逐条比对执行，真正把增量写入 `data/csv/`，重跑 `tools/csv_to_json.py` 生成 `site/data/*.json`，`tools/validate.py` 再过一遍，一并提交。合入者对史料判断没有自由裁量权——落实不了或存疑一律停下报告，不得自行改动史料口径来"让它能合"。
- **两者可以跨轮次错开**：备料完成不代表下一轮必须合入（round13 楚秦线即为一例：v1.7 成文时该批已备料完成但暂不合入，v1.9 随 fix13 一并正式合入，见 `data/incoming/round13/CHANGES.md`），是否合入、何时合入由领队按整体节奏排期决定，`data/incoming/` 下的目录可以合法地"等待"多轮。
- **合入前置条件的显式化**：若某批增量依赖尚未落地的基础设施（如本轮楚秦线的雍/郢坐标依赖底图西扩，见 `docs/map_expansion_plan.md`），该依赖必须在 `CHANGES.md` 中显式注明，合入者据此判断是否具备合入条件，不得"先合数据、坐标留空将来再补"之外的隐性依赖被忽略。
