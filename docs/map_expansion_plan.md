# 底图西扩（含南扩）评估方案

状态：**评估稿，未执行**。本方案不改动 `site/assets/map/base_map.svg`、`site/app.js`、`tools/validate.py`、`docs/conventions.md` 任何一行；仅作可行性验算与执行路线设计，供领队与 Xiangtao 核准后另行派工执行。

背景：`data/incoming/round13`（秦穆公／楚成王两主角线）已产出 **L_YONG（雍，秦都）** 与 **L_YINGDU（郢，楚都）** 两地，因经度低于现底图东经 110 下限（雍约 107.4E）、纬度低于现底图北纬 32 下限（郢约 30.35N），round13 依裁定"坐标留空待西扩/待南扩"。本方案评估是否、以及如何一次性将底图范围西扩且南扩以纳入秦楚腹地。

## 一、新投影参数验算

### 1.1 现行公式（conventions.md §4，v1.6）

```
覆盖范围：东经 110.0–122.0，北纬 32.0–38.5
viewBox="0 0 1200 700"
x = (lng - 110.0) / 12.0 * 1200
y = 700 - (lat - 32.0) / 6.5 * 700
```

### 1.2 领队起点方案（供校验）

```
覆盖范围：东经 105.0–122.0，北纬 30.0–38.5
viewBox 不变 "0 0 1200 700"
x = (lng - 105.0) / 17.0 * 1200
y = 700 - (lat - 30.0) / 8.5 * 700
```

注：此方案名为"西扩"，实为**西扩 + 南扩同步**（经度下限 110→105，纬度下限 32→30），一次覆盖雍（西界问题）与郢（南界问题）两处缺口，命名与实际改动范围不完全对应，建议正式文档中改称"西南扩"或"腹地扩边"以免歧义。

### 1.3 四点验算（四舍五入到整数像素）

| 地点 | lng, lat | 新 x, y | 现行 x, y（参照） |
|---|---|---|---|
| 雍（秦都） | 107.4, 34.5 | **169, 329** | 超界，现行公式无法计算（x 为负） |
| 郢（楚都） | 112.2, 30.35 | **508, 671** | 超界，现行公式无法计算（y 超出 700 下方） |
| 临淄 | 118.31, 36.83 | **940, 138** | 831, 180 |
| 曲阜 | 116.99, 35.60 | **846, 239** | 699, 312 |

验算过程：
- 雍：x=(107.4-105)/17×1200=169.4；y=700-(34.5-30)/8.5×700=329.4
- 郢：x=(112.2-105)/17×1200=508.2；y=700-(30.35-30)/8.5×700=671.2
- 临淄：x=(118.31-105)/17×1200=939.5；y=700-(36.83-30)/8.5×700=137.6
- 曲阜：x=(116.99-105)/17×1200=846.4；y=700-(35.60-30)/8.5×700=238.8

四点均落在 [0,1200]×[0,700] 以内，公式本身可行。

**边距提醒（风险点，非否决项）**：
- 雍距西边界约 2.4°（≈240 km 图上留白），边距充裕。
- **郢距南边界仅约 0.35°（≈39 km 图上留白，像素上仅 671→700 共 29px）**，裕量很薄。若后续有比郢更靠南的地点（如更南的楚地、或吴越相关地点），30.0N 下限会不够用，需二次改投影。建议领队核准前确认："30.0N 下限是否已按当前已知最南地点留有安全边际"，或酌情把下限压到 29.0–29.5N 换取余量（viewBox 与 x 公式不受影响，仅 y 公式常数变动）。本方案不擅自改动领队给出的 30.0 起点，仅作风险提示。

### 1.4 纵横比例失真评估

地图是等距圆柱式简易投影（经度、纬度各自线性映射到像素，未做真实地图投影的纬度校正），失真程度用「x 方向 px/km 与 y 方向 px/km 之比」衡量，1.0 为无失真：

```
失真比 = (px/度经) / [(px/度纬) × cos(纬度)]
```

（cos(纬度) 把"度经"换算为该纬度上的实际公里数；1 度纬≈111.32 km 视为常量。）

| 方案 | px/度经 | px/度纬 | 纬度 32°处失真比 | 纬度 38.5°处失真比 |
|---|---|---|---|---|
| 现行（110–122E,32–38.5N） | 100.0 | 107.7 | 1.095（+9.5%） | 1.187（+18.7%） |
| 新方案（105–122E,30–38.5N） | 70.6 | 82.4 | 0.990（-1%，30°处） | 1.095（+9.5%，38.5°处） |

结论：**新方案的纵横失真不但没有恶化，反而全区间收窄**（现行 +9.5%~+18.7% → 新方案 -1%~+9.5%）。原因是经度跨度从 12° 扩到 17°（+42%）而 x 像素不变（仍 1200px），纬度跨度从 6.5° 扩到 8.5°（+31%）而 y 像素不变（仍 700px），两者放大倍率不同，恰好抵消了原公式里"经度方向像素密度过高"的问题。**评估结论：新参数在失真上是可接受甚至是改进的，无需另调 viewBox 或另设分段投影。**

站内本就注明"地图只是示意……非考据疆界"（`site/index.html` 关于页），个位数到十位数百分比的示意性失真在此语境下不构成问题。

### 2.1 base_map.svg 重绘范围（关键发现：可用仿射变换复用旧图，不必整图重画）

现行公式与新公式都是"经纬度各自独立线性映射到像素"，没有旋转/斜切，因此旧像素坐标到新像素坐标之间也是一个纯粹的**缩放+平移**（仿射变换），可以解析求出：

```
x_new = 0.70588 × x_old + 352.94   （即 12/17 的缩放 + 105→110 差值平移）
y_new = 0.76471 × y_old            （即 6.5/8.5 的缩放，原点不动）
```

（推导：把旧公式反解出 lng/lat 代入新公式即得上式，系数 12/17=0.70588、6.5/8.5=0.76471 与表 1.4 的 px/度 比值一致。）

**含义**：`layer-sea` / `layer-states` / `layer-rivers` / `layer-mountains` 四组现有手绘 path，理论上可以**整体套一层 `<g transform="matrix(0.70588,0,0,0.76471,352.94,0)">`** 而不必逐笔重新画——原有齐、鲁、宋、卫等 11 国色块与水系会被正确地缩小并平移到与新 `project()` 输出一致的位置，省去这部分的重绘工作量。

但有两个例外，仍需 Vision 手工处理：
- **`layer-labels`（11 个国名文字）**：若随 `<g>` 一起做非均匀缩放（x 缩 0.706、y 缩 0.765），字形会被压扁变形。建议改为对每个文字的 x,y 锚点单独套用上述公式重算坐标，`font-size` 保持不变或略增（因为地图实际展现的地理范围变大了 42%×31%，全图观感"缩小"，建议 17px→19px 左右做视觉补偿），文字本身不进 transform 分组。
- **新增内容（无法靠变换生成，需新画）**：
  - 秦岭示意（山地记号，风格比照现有 `layer-mountains` 的折线记号）
  - 渭水（秦地水系，东西向注入河水故道方向）
  - 汉水（楚地水系，经楚境南下）
  - 江淮示意（南界新增区域的粗略水系暗示，非考据河道，仅为"再往南是江淮"的方向感）
  - 秦、楚两色块（比照现有 11 国 ellipse/path 风格新增两个 `data-state` 色块，供首页地图导航热区复用同一套点击绑定逻辑）
  - 上述新内容画在**新坐标系**下（即直接按新 `project()` 输出的像素作图，不进那个仿射变换 `<g>`，避免二次变换搞乱换算）

结论：**"重绘"的真实工作量小于"整图重画"，是"旧图仿射复用 + 5 类新增元素手绘 + 标注层坐标重算"的组合**，仍建议按"中高复杂度、需 Vision 执笔"排期，但可以据此给出更准的工时估算依据。

### 2.2 既有 62 地点重投影核验法

新范围 [105-122E,30-38.5N] 是旧范围 [110-122E,32-38.5N] 的严格超集，因此：
- **数值合法性**：62 地点现有 lat/lng（含 2 条空值）无需任何改动即自动满足新 `validate.py` 边界检查，机械层面零风险。
- **像素位置**（需要人工/半自动核验的部分）：所有点的屏幕像素坐标都会按 2.1 节仿射公式整体平移缩放。核验方法：
  1. 写一个一次性小脚本（可用 Python 标准库，不入 `tools/`，用完即弃或放 `data/incoming/` 之外的临时位置），对 `data/csv/places.csv` 全部 60 条有坐标的行，同时按旧公式与新公式各算一遍像素坐标，核对 `新x,新y` 是否等于对旧坐标套用 2.1 节仿射公式的结果——两者理论上应逐点相等（互为交叉验证，能抓出 `app.js` 新常量写错的低级错误）。
  2. 抽样目视核验：临淄、曲阜、新郑、雍、郢等已知地标在重绘后的 `base_map.svg` 上打开浏览器实测，确认点位仍落在对应国别色块内部（不是核验数值，是核验"艺术图"与"数据图"是否对齐）——此步骤需在 base_map.svg 重绘完成后由 Vision 执行，Skipper 侧只能做数值交叉验证。
  3. 全部通过后，`tools/validate.py` 正常跑一遍作为最终机械把关（预期本轮因无数据改动而 100% 通过，仅确认常量改动没有引入语法错误）。

## 三、执行建议（与楚秦数据合入同轮/第13轮执行）

建议拆分为两个提交批次，之间设一个可回滚检查点，理由：第①批是纯机械改动（文档+校验器常量），可独立验证、独立回滚，不影响任何已部署页面的视觉呈现；第②批（app.js 常量 + base_map.svg 重绘）必须**原子上线**——如果只改 `project()` 常量不改底图，或只改底图不改 `project()`，线上会出现"点位与色块错位"的显式 bug，因此这两个文件必须在同一个 commit 里一起改、一起过审、一起部署。

**第①批（Skipper 执行，独立可回滚）**
1. 改 `docs/conventions.md` §4：新范围、新公式、四点校验样例（本方案 1.3 节内容原样搬入，含边距提醒）。
2. 改 `tools/validate.py` 的 `LNG_MIN/LNG_MAX/LAT_MIN/LAT_MAX` 四个常量。
3. 跑 `python tools/validate.py`，预期 exit 0（现有数据是新范围的子集，理论零报错）。
4. 单独提交，commit message 注明"仅扩大坐标合法区间，地图视觉未变，本批次单独可回滚"。
5. **回滚点 A**：若发现范围数字有误，`git revert` 此提交即完全恢复，不影响其他任何文件。

**第②批（Vision 执笔 base_map.svg，Skipper 配合改 app.js，二者合一提交）**
1. Vision 按 2.1 节方案重绘 `base_map.svg`（仿射复用旧图层 + 标注层坐标重算 + 5 类新增内容手绘）。
2. Skipper 同步改 `site/app.js` 的 `project()` 四常量（105/17/30/8.5）。
3. 按 2.2 节做数值交叉验证 + 目视抽样核验。
4. 本地/预览环境实测：首页地图导航、人物地图视图、分享卡生成器（确认自动拾取新底图）三处功能回归，四档常用视口（360/390/414/680，沿用 r13/r14 已建立的 CDP 实测习惯）无横向溢出、无点位错位。
5. 二者合一提交，push，等 GitHub Pages / Cloudflare Pages 部署完成后线上复测同一批用例。
6. **回滚点 B**：若线上出现点位错位或视觉问题，`git revert` 此单一提交，`base_map.svg` 与 `app.js` 一起退回旧版本，站点回到"雍/郢留空、旧范围"的已知良好状态，不影响第①批已经生效的校验器扩容（新数据仍能过校验，只是地图暂不显示秦楚腹地）。

**第③批（Sophia 主导，不属本方案范围，仅列作衔接提醒）**
- round13 楚秦数据正式合入 `data/csv/` 时，`fixes_people.csv` 及 `L_YONG`/`L_YINGDU` 坐标回填应等第②批底图上线后再做，避免"数据已填坐标、底图还没扩"的中间态导致点位画出图外。

## 四、结论

- 领队起点方案（105-122E,30-38.5N；viewBox 不变）**验算通过**，四点坐标均在界内，纵横失真较现行公式**不升反降**（全区间失真从 +9.5%~+18.7% 收窄至 -1%~+9.5%）。
- 唯一风险点：郢距南边界仅约 39km 图上余量，偏薄，建议领队核准前二次确认是否需要把纬度下限从 30.0 再压低半度左右留余量；本方案不擅改领队给出的起点数字。
- `base_map.svg` 的"重绘"经分析可拆为"仿射变换复用旧图层（低风险、可脚本辅助）+ 5 类新内容手绘（中高复杂度，Vision 主责）+ 标注层坐标重算（低风险）"，工作量小于外界直觉的"整图重画"。
- `og-card.png` 确认不含地图内容，**本次扩边无需更新**；人物分享卡因运行时抓取 SVG 会自动跟随，只需回归测试一次。
- 建议拆两批提交：第①批（conventions+validate 常量）独立可回滚；第②批（app.js+base_map.svg）必须原子提交，各设明确回滚点。第③批（round13 楚秦数据实际合入）应在第②批稳定上线后再做，避免中间态。

本方案未改动任何代码/数据/校验器文件，请领队/Xiangtao 核准后另行派工执行。
